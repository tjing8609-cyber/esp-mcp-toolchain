# ADR 0002：后台串口 Monitor 架构

- 状态：Accepted
- 日期：2026-07-13

## 背景

固定时长串口捕获不能持续观察设备。后台 Monitor 还必须处理工程切换、连续读取语义、缓冲区上限、二进制输出、进程退出和多个 MCP 进程争用同一串口。

## 决定

### 不可变绑定

`esp_serial_monitor_start` 在启动时捕获 `project_id`、`workspace_root`、日志根目录、端口身份和串口参数。Worker 之后不读取全局当前项目，活动项目切换不会改变已有会话的写入位置。

### 状态机

会话状态为 `STARTING`、`RUNNING`、`STOPPING`、`STOPPED`、`FAILED`、`DISCONNECTED`。状态转换写入 manifest；串口断开与内部失败分开报告。

### 读取协议

接口为：

```text
esp_serial_monitor_read(
    run_id,
    after_seq=None,
    max_bytes=65536,
    wait_ms=0,
    representation="text",
)
```

- 每个记录有单调递增 `seq`，调用方用 `after_seq` 读取之后的数据。
- 连续调用相同游标会得到相同的仍在缓冲区内的数据；使用返回的 `next_after_seq` 才向前推进。
- 缓冲区有界，旧记录被淘汰时返回 `dropped_before_seq`。
- 记录按串口读取块而不是按行生成，不等待换行。
- 原始字节始终保留；text 使用增量 UTF-8 解码并替换无法解码的字节，base64 可无损表示二进制数据。
- `max_bytes` 限制单次返回，`wait_ms=0` 立即返回，无新数据时可选择有界等待。

### 存储与配额

- 活动块先写 `.part`，完成后原子改名为 `.bin` 并记录 SHA-256。
- 默认环形缓冲 1 MiB、块大小 8 MiB、单会话原始日志 256 MiB、项目原始日志安全上限 2 GiB，可通过环境变量覆盖。
- 当前项目上限是启动时项目占用加本会话预计写入的保护，不宣称多个独立进程并发写入时具有全局原子配额。

### 锁与端口身份

- 进程内同一端口只允许一个 Monitor；跨进程使用带 PID 和进程创建标记的锁文件。
- manifest 和锁都记录 PID、进程创建标记与本次进程 token。恢复前先确认原所有进程已经结束，不能把另一个仍在运行的 MCP 会话标成陈旧。
- 下次启动会回收可证明进程已结束的陈旧锁；串口能否实际打开仍由操作系统和 pyserial 最终裁定。
- 保存 device path、VID/PID、序列号、location、描述等可用身份字段，不只依赖 `COM` 名称。

### 退出与限制

- 正常 shutdown、stdin EOF、可捕获信号、`atexit` 和内部异常执行有界清理。
- 强制终止、崩溃或断电时清理代码可能没有机会执行；此时依赖操作系统释放句柄和下次启动清理陈旧状态。
- 打开串口前尽量设置 DTR/RTS 为非激活默认值，但不同 USB-UART 电路可能仍影响 EN/GPIO0，不能承诺所有板卡都不会复位。

## 验证

假串口和进程级测试覆盖正常退出、stdin EOF、强制终止恢复、跨进程冲突、断连、磁盘故障、配额、分块、UTF-8 边界和二进制读取。真实 ESP USB 串口在本次验证时未枚举，因此硬件门禁仍未通过。
