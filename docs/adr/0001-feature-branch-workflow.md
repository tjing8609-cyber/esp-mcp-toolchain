# ADR 0001：main / test 双工作树开发流程

- 状态：Accepted
- 日期：2026-07-13

## 背景

项目已经有 `index` 和 `index-test` 两个工作树。用户要求产品实现提交到 `main`，测试文件和测试目录只在 `test` 分支增加；同时任何新增功能或修复都必须先通过全量测试。流程必须防止两个分支漂移，也不能为了方便把测试提交重新带回 `main`。

## 决定

- `index` 工作树承载 `main`，只提交产品实现和必要文档。
- `index-test` 工作树承载 `test`；该分支自己的提交只修改测试文件、测试目录和验证规则。
- 开始测试工作前先把最新 `main` 同步到 `test`，但不把 `test` 的分支专属测试合回 `main`。
- 修复 bug 时先在 `test` 写出能够稳定复现问题的红灯测试，再让测试工作树通过 `PYTHONPATH` 或等价的明确路径加载 `index` 中待提交源码。
- 跨工作树全量测试通过后才提交 `main`；随后把新的 `main` 同步到 `test`，提交测试并再次运行全量测试和 CI。
- 依赖真实硬件的能力还要通过目标板卡门禁；未验证状态必须写入 README 的未完成或风险部分。

## 结果

实现和测试的所有权边界清晰，`main` 不会因测试实验产生额外文件；代价是门禁必须明确记录两个工作树的提交和源码加载路径，并在 `main` 更新后及时同步 `test`，否则测试结果不能代表当前主线。
