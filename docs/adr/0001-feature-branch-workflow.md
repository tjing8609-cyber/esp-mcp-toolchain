# ADR 0001：功能分支开发流程

- 状态：Accepted
- 日期：2026-07-13

## 背景

旧流程把测试长期维护在 `test` 分支、实现维护在 `main`。两个分支逐渐漂移后，同一时刻可能出现主线只有旧测试、测试分支只有新契约的情况，无法用一个提交证明“实现与测试共同通过”。

## 决定

- 每项功能从最新 `main` 创建 `feature/<topic>`。
- 实现、测试和必要文档放在同一功能分支。
- 本地全量测试和 GitHub Actions 都通过后才允许合入。
- 依赖真实硬件的能力还要通过硬件门禁。
- 历史 `test` 分支保留以便追溯，但不再接收新开发。

## 结果

一个功能提交可以同时回答“实现是什么、测试覆盖什么、当前门禁是否满足”。代价是每个功能分支要及时从 `main` 更新，并在提交前运行完整测试矩阵。
